# CVI Project Structure

## File Types

### `.prj` — Project File

Text-based file that defines a CVI project. Contains:
- List of source files (`.c`) included in the project
- Compiler options (include paths, preprocessor defines)
- Linker options (libraries, instrument drivers)
- Target configuration (executable, DLL, static library)
- Build settings (debug/release, 32/64 bit)

**How to read it:** Open as text. Look for `[Source Files]` section for the list of `.c` files. Look for `[Include Paths]` for dependencies. Look for `[Instrument Driver Files]` for hardware driver dependencies.

### `.cws` — Workspace File

Text-based file that groups multiple `.prj` files into a workspace. Analogous to a Visual Studio `.sln` file.

**How to read it:** Open as text. Look for `[Project List]` section for the list of `.prj` files.

### `.uir` — User Interface Resource

**Binary file** — NOT directly readable. Defines the UI: panels, controls (buttons, graphs, numeric inputs, rings, LEDs, text boxes), menus, and their properties.

**What it generates:** Each `.uir` file has a corresponding `.h` header file (auto-generated by CVI IDE) containing:
- Panel constant: `#define PANEL 1`
- Control constants: `#define PANEL_BUTTON_START 2`, `#define PANEL_GRAPH_DATA 3`, etc.
- Menu constants: `#define MENUBAR 1`, `#define MENUBAR_FILE 2`, etc.

**Strategy for analysis:** Since `.uir` is binary, analyze via:
1. **Read the generated `.h` file** — lists all control IDs and their names
2. **Search for callback registrations** — `InstallCtrlCallback(panel, PANEL_CONTROL, CVICALLBACK FuncName, 0)`
3. **Search for `CVICALLBACK` functions** — they handle UI events

### `.fp` — Function Panel File

Text-based file that documents function prototypes for instrument drivers. Contains function names, parameter descriptions, help text. Used by CVI IDE for auto-complete and function panels.

### `.sub` — Subproject File

Used in older CVI versions to define sub-components. Less common in modern CVI.

## Callback Naming Convention

CVI uses a naming convention for UI callbacks:

```c
int CVICALLBACK PanelName_ControlName_Callback(
    int panel,
    int control,
    int event,
    void *callbackData,
    int eventData1,
    int eventData2)
```

**Common variations:**
- `MainPanel_StartBtn_CB` — abbreviated
- `MAIN_STARTBTN_CALLBACK` — uppercase
- `OnStartButtonPressed` — custom naming (less common)
- `panelCB` — generic per-panel callback

**The naming convention is NOT enforced** — some codebases use consistent naming, others are chaotic. The reliable way to find callbacks is to search for `CVICALLBACK` in `.c` files.

## Entry Point Pattern

Typical CVI application startup:

```c
int main(int argc, char *argv[])
{
    int panelHandle;

    // Initialize CVI runtime
    if (InitCVIRTE(0, argv, 0) == 0)
        return -1;

    // Load UI from .uir file
    panelHandle = LoadPanel(0, "myapp.uir", PANEL);
    if (panelHandle < 0)
        return -1;

    // Display the main panel
    DisplayPanel(panelHandle);

    // Enter event loop — blocks until QuitUserInterface() is called
    RunUserInterface();

    // Cleanup
    DiscardPanel(panelHandle);
    CloseCVIRTE();
    return 0;
}
```

**Variations:**
- Multiple panels loaded before `RunUserInterface()`
- Hardware initialization between `LoadPanel` and `RunUserInterface`
- `LoadPanelEx` with callback map instead of `InstallCtrlCallback`
- Timer callbacks started before entering event loop

## Module Organization

CVI has no formal module system. Developers organize code by convention:

**Common patterns:**
- One `.c`/`.h` pair per instrument (e.g., `sr830.c`/`sr830.h`, `keithley2000.c`/`keithley2000.h`)
- One `.c` for main application logic + UI callbacks
- One `.c` for data processing/analysis
- One `.c` for file I/O
- Shared utility functions in a `utils.c` or `common.c`

**God file anti-pattern:** Everything in one massive `.c` file (common in smaller or older projects).

**How to identify modules:** Look at `#include` chains. Files that `#include` each other's headers form an implicit module. Files that share global variables are tightly coupled even without includes.
